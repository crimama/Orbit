# /translate

**사용법**: `/translate <UI/UX 문제 또는 개선사항>`

사용자가 느끼는 UI/UX 또는 프론트엔드 디자인 측면의 문제·개선사항을 받아서, **분석 → 문제 도출 → 구현**까지 한 번에 수행한다.

## 동작 흐름

### Phase 1: 문제 분석

1. 사용자 설명에서 **어떤 화면/컴포넌트**에서 **어떤 문제**를 느끼는지 파악
2. 해당 컴포넌트의 현재 코드를 Read하여 **실제 구현 상태** 확인
3. 문제의 근본 원인 도출:
   - 레이아웃/공간 문제 (너무 큼, 너무 작음, 정렬 불량)
   - 데이터 표시 문제 (안 나옴, 잘못된 값, 형식 오류)
   - 인터랙션 문제 (클릭 안 됨, 반응 없음, 잘못된 동작)
   - 스타일 문제 (색상, 폰트, 간격, 다크테마 불일치)
   - 데이터 흐름 문제 (API 미연결, 소켓 이벤트 누락, 상태 미전파)

### Phase 2: 해결 방안

근본 원인에 따라 최소 영향 범위의 수정안 제시:
- 프론트엔드만으로 해결 가능한지
- API/백엔드 수정이 필요한지
- 새 컴포넌트가 필요한지, 기존 컴포넌트 수정으로 충분한지

### Phase 3: 구현

수정안을 **즉시 구현**한다:
1. 코드 수정 (Edit/Write)
2. 타입 체크 (`npx tsc --noEmit`)
3. 빌드 확인 (`npm run build` — Error 체크)
4. 커밋 & 푸시

## 규칙

- **분석만 하지 말고 바로 고친다** — 사용자에게 "수정할까요?" 묻지 않고 직접 구현
- **현재 코드를 반드시 먼저 읽는다** — 추측으로 수정하지 않음
- **최소 변경 원칙** — 문제 해결에 필요한 최소한의 코드만 수정
- **기존 스타일 유지** — 프로젝트의 다크 테마, Tailwind 패턴, 컴포넌트 구조 유지
- **데이터 문제는 API까지 추적** — 프론트에서 안 나오면 API 응답, 서버 로직, DB까지 확인
- **빌드 깨뜨리지 않음** — 수정 후 반드시 tsc + build 확인
