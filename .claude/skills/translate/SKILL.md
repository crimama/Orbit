# /translate

**사용법**: `/translate <UI/UX 설명>`

UI/UX 관점의 설명을 받아서, 실제로 수정/생성해야 할 코드 요소들을 매핑해주는 번역 레이어.

## 동작

### 0. 부트스트랩 (COMPONENT_MAP.md 없을 때)

COMPONENT_MAP.md가 존재하지 않으면 **자동 생성 모드** 진입:

1. Explore 에이전트를 사용하여 프로젝트 코드베이스 전체 스캔:
   - `src/components/` (또는 프론트엔드 컴포넌트 디렉토리) → 모든 컴포넌트 + props + 자식 관계
   - `src/app/api/` (또는 API 라우트 디렉토리) → 모든 엔드포인트 + 메서드
   - `src/server/` (또는 백엔드 디렉토리) → 모든 서버 모듈 + 주요 메서드
   - Socket/WebSocket 이벤트 계약 (있는 경우)
   - DB 모델 (Prisma/TypeORM/Sequelize 등)
   - 공유 훅/유틸리티
2. 스캔 결과를 COMPONENT_MAP.md 형식에 맞춰 `.claude/skills/translate/COMPONENT_MAP.md`에 작성
3. 사용자에게 "COMPONENT_MAP.md를 생성했습니다. 확인 후 수정할 부분이 있으면 알려주세요." 안내
4. 부트스트랩 완료 후 아래 1번부터 정상 진행

> **참고**: 프로젝트 구조가 크게 바뀌었을 때 `/translate --rescan`으로 COMPONENT_MAP.md를 재생성할 수 있음.

---

1. `$ARGUMENTS`에서 UI/UX 설명 파싱
   - 인자 없으면 → "어떤 UI/UX 변경을 설명해주세요" 안내 후 종료
   - `--rescan` 플래그 → 기존 COMPONENT_MAP.md 삭제 후 부트스트랩 재실행

2. **COMPONENT_MAP.md 로드**: `.claude/skills/translate/COMPONENT_MAP.md` 읽기

3. **키워드 분석**: 사용자 설명에서 UI 영역 키워드 추출
   - 예: "터미널", "사이드바", "세션 목록", "그래프", "파일 편집", "모바일", "승인 팝업" 등
   - 각 키워드를 COMPONENT_MAP.md의 매핑 테이블과 대조

4. **영향 범위 매핑**: 다음 4개 레이어를 순서대로 분석하여 출력

   ### a. 프론트엔드 (수정할 컴포넌트)
   - 직접 관련된 컴포넌트 파일 경로
   - 해당 컴포넌트의 주요 props
   - 부모/자식 컴포넌트 관계 (데이터 흐름)
   - 필요하다면 실제 파일을 Read하여 현재 구조 확인

   ### b. API 레이어 (호출/수정할 엔드포인트)
   - 관련 API 라우트 파일
   - HTTP 메서드 및 요청/응답 형태
   - 새 엔드포인트가 필요한 경우 제안

   ### c. 서버 모듈 (백엔드 로직)
   - 관련 서버 매니저/핸들러
   - 수정이 필요한 메서드
   - 새 모듈이 필요한 경우 제안

   ### d. 실시간 통신 (Socket.io 이벤트)
   - 관련 기존 이벤트
   - 새 이벤트가 필요한 경우 이벤트명 + 페이로드 제안
   - 관련 훅 (useSocket, usePendingApprovals 등)

   ### e. 데이터 (Prisma 모델/타입)
   - 관련 Prisma 모델
   - 새 필드/모델 필요 시 제안
   - `src/lib/types.ts` 타입 변경 필요 여부

5. **개발 체크리스트 출력**: 위 분석을 바탕으로 구체적 작업 목록 생성

## 출력 형식

```markdown
## 🔍 번역 결과: {사용자 설명 요약}

### 프론트엔드
| 파일 | 변경 내용 | 난이도 |
|------|----------|--------|
| `src/components/...` | ... | 낮음/중간/높음 |

### API
| 엔드포인트 | 변경 내용 |
|-----------|----------|
| `/api/...` | ... |

### 서버
| 모듈 | 변경 내용 |
|------|----------|
| `src/server/...` | ... |

### Socket.io
| 이벤트 | 변경 내용 |
|--------|----------|
| `event-name` | ... |

### 데이터
| 모델/타입 | 변경 내용 |
|----------|----------|
| `Model` | ... |

### 개발 순서
1. [ ] ...
2. [ ] ...
```

## 규칙

- **COMPONENT_MAP.md 참조 필수**: 맵에 없는 요소는 실제 파일을 Read/Grep하여 확인
- **구체적 경로 제시**: "컴포넌트를 수정하세요"가 아니라 정확한 파일:라인 제시
- **데이터 흐름 명시**: props 전달 경로, Socket 이벤트 흐름, API 호출 체인을 명시
- **과잉 매핑 금지**: 관련 없는 요소를 끌어들이지 않는다. 최소 영향 범위만 매핑
- **불확실하면 확인**: 매핑이 불확실하면 실제 코드를 읽어서 검증
- 이 스킬의 출력 후 사용자가 구현을 원하면 → `/feature` 또는 직접 구현 진행
